Credentials is the primitive that lets employees safely hold secrets. Naive provides encrypted storage for API keys, passwords, tokens, and any other secret value an employee needs to authenticate against an external system. Credentials are encrypted at rest, scoped to the company so multiple employees can share them, and every access is logged for audit compliance — the same way a real company uses a password manager with role-based access.

## Tools

| Tool | Type | Description |
|------|------|-------------|
| `store_credential` | Extended | Securely store credentials for a service |
| `get_credential` | Extended | Retrieve stored credentials by reference |
| `refresh_identity` | Extended | Re-generate identity files after credential changes |

## Storing Credentials

Use `store_credential` to save credentials for a service. You get back a reference URI for retrieval.

```bash
curl -X POST https://api.usenaive.ai/v1/credentials/store \
  -H "Authorization: Bearer nv_sk_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "openai",
    "credentials": {
      "api_key": "sk-proj-abc123..."
    }
  }'
```

**Response:**

```json
{
  "ok": true,
  "ref": "paperclip://secrets/agent-credential/agent_abc/openai"
}
```

Save the `ref` — you'll use it to retrieve the credentials later.

### Parameters

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | Yes | Service name (e.g., `github`, `stripe`, `openai`) |
| `credentials` | object | Yes | Key-value pairs to store |

The `credentials` object is flexible — store whatever key-value pairs make sense for the service:

```json
{
  "service": "aws",
  "credentials": {
    "access_key_id": "AKIA...",
    "secret_access_key": "wJalr...",
    "region": "us-east-1"
  }
}
```

## Retrieving Credentials

Use `get_credential` with the reference URI:

```bash
curl -X POST https://api.usenaive.ai/v1/credentials/resolve \
  -H "Authorization: Bearer nv_sk_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "ref": "paperclip://secrets/agent-credential/agent_abc/openai"
  }'
```

**Response:**

```json
{
  "secret_id": "sec_xyz789",
  "name": "agent-credential/agent_abc/openai",
  "credentials": {
    "api_key": "sk-proj-abc123..."
  }
}
```

<Info>
Every credential retrieval is logged to the activity log. This creates a full audit trail of which agent accessed which secrets and when.
</Info>

## Refreshing Identity

After storing new credentials, getting a phone number, or registering a domain, call `refresh_identity` to update the agent's identity files:

```bash
curl -X POST https://api.usenaive.ai/v1/credentials/refresh-identity \
  -H "Authorization: Bearer nv_sk_your_key"
```

**Response:**

```json
{
  "identity": "# Agent Identity\n...",
  "accounts": "# Connected Accounts\n- OpenAI: paperclip://secrets/agent-credential/agent_abc/openai\n...",
  "profile": "# Agent Profile\n..."
}
```

This regenerates three scaffold files:
- **identity.md** — Agent identity and capabilities
- **accounts.md** — Connected accounts with credential references
- **profile.md** — Focus, context, and strategy

## Security Properties

<CardGroup cols={2}>
  <Card title="Encrypted at rest">
    All credentials are encrypted before being written to the database
  </Card>
  <Card title="Audit logged">
    Every credential access is recorded with agent ID, timestamp, and reference
  </Card>
  <Card title="Company-wide access">
    Credentials are scoped to the company — any agent in the company can access them
  </Card>
  <Card title="Reference-based">
    Credentials are referenced by URI, not stored in config files or environment variables
  </Card>
</CardGroup>

## Error Handling

| Error | Cause | Recovery |
|-------|-------|----------|
| `ref_not_found` | Credential reference doesn't exist | Check the ref URI and try again |
| `service_exists` | Credentials already stored for this service | Use a different service name or update the existing entry |
| `invalid_ref` | Malformed credential reference URI | Use the exact ref returned from `store_credential` |

## Best Practices

<Steps>
  <Step title="Never store raw secrets in files">
    Use `store_credential` instead of writing API keys to `.env` files or config. The credential reference (`paperclip://...`) is safe to store in code and config.
  </Step>
  <Step title="Use descriptive service names">
    Name services clearly: `openai`, `stripe-production`, `github-acme-org`. This makes the accounts list readable.
  </Step>
  <Step title="Refresh identity after changes">
    Call `refresh_identity` after storing credentials, claiming a phone number, or registering a domain. This keeps the agent's context up to date.
  </Step>
  <Step title="Store purchase credentials via transactions">
    When a purchase produces login credentials, store them in `log_card_transaction` with `credential_data` for a complete purchase-to-access audit trail. Use `store_credential` for long-term storage.
  </Step>
</Steps>

## Typical Workflow

```
Task: "Set up the OpenAI API key I just got"

1. store_credential({
     service: "openai",
     credentials: { api_key: "sk-proj-..." }
   })
   → ref: "paperclip://secrets/agent-credential/agent_abc/openai"

2. refresh_identity()
   → Identity files updated with new credential reference

Later, when the agent needs the key:

3. get_credential({
     ref: "paperclip://secrets/agent-credential/agent_abc/openai"
   })
   → { api_key: "sk-proj-..." }
   → Access logged to audit trail
```
