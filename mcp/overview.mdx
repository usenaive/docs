The Naive MCP server is the bridge between an agent harness (Claude Desktop, Cursor, Hermes, your own runtime) and the **primitives layer** — the tools every Naive employee uses to act on the world. One connection gives an agent access to email, social media, image and video generation, web search, payments, browser automation, and 100+ third-party integrations.

If you're using the full Naive Core stack, Hermes already has this wired up for every employee. If you're connecting your own agent, the MCP server is how you plug into the same primitives Hermes-driven employees use.

## Architecture

Every primitive call flows through a simple chain from the agent harness to Naive:

```
Agent Harness (Claude, Cursor, Hermes, custom)
        │
        │  MCP protocol (stdio)
        ▼
    naive-mcp
        │
        │  HTTPS + Bearer token  (employee-scoped)
        ▼
  api.usenaive.ai           ← Primitives API
        │
        ▼
  Real-world services       ← email, bundle.social, Stripe Issuing, etc.
```

The MCP server runs as a local subprocess launched by your harness. It communicates over **stdio** using the [Model Context Protocol](https://modelcontextprotocol.io), translating tool calls into authenticated HTTPS requests against the Naive Primitives API. Identity (which employee, which company) comes from the API key — Core handles the rest.

<Note>
  The MCP server never stores data locally. All state — email, social accounts, credentials, billing — lives on the Naive platform and is scoped to your API key.
</Note>

## Tool Registration Strategy

The server registers **12 tools** in your agent's context window to keep it lean and focused:

<CardGroup cols={2}>
  <Card title="8 Core Tools" icon="star">
    Registered with full Zod schemas. Your agent calls them directly — no discovery step needed. These cover the highest-frequency actions: email, social media, images, search, stock media, and integrations.
  </Card>
  <Card title="4 Meta-Tools" icon="compass">
    `naive_context`, `naive_search`, `naive_tool_help`, and `naive_execute` let your agent discover and use all **54 tools** in the catalog without bloating the context window.
  </Card>
</CardGroup>

This two-tier approach means your agent always has the most useful tools at hand, while the full catalog remains accessible on demand.

## Getting Started

<Steps>
  <Step title="Install the SDK">
    ```bash
    npm install -g @usenaive/sdk
    ```
  </Step>
  <Step title="Run the setup wizard">
    ```bash
    naive init
    ```
    This validates your API key, writes the MCP config for your agent runtime, and saves settings to `~/.naive/config.json`.
  </Step>
  <Step title="Restart your agent runtime">
    Restart Claude Desktop or Cursor to load the new MCP server configuration.
  </Step>
  <Step title="Start using tools">
    Ask your agent to perform a real-world action:

    > "Send an email to alice@example.com about our product launch"

    The agent will call `send_email` through the MCP connection.
  </Step>
</Steps>

## Starting the Server

In most cases, your agent runtime launches the MCP server automatically based on the config written by `naive init`. You can also start it manually:

```bash
NAIVE_API_KEY=nv_sk_... naive mcp start
```

Or configure it directly in your runtime's MCP config:

<CodeGroup>
```json Claude Desktop
{
  "mcpServers": {
    "naive": {
      "command": "naive",
      "args": ["mcp", "start"],
      "env": {
        "NAIVE_API_KEY": "nv_sk_..."
      }
    }
  }
}
```

```json Cursor
{
  "mcpServers": {
    "naive": {
      "command": "naive",
      "args": ["mcp", "start"],
      "env": {
        "NAIVE_API_KEY": "nv_sk_..."
      }
    }
  }
}
```
</CodeGroup>

## Environment Variables

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `NAIVE_API_KEY` | Yes | — | Your Naive API key (starts with `nv_sk_`) |
| `NAIVE_API_URL` | No | `https://api.usenaive.ai` | API base URL (override for local development) |

<Warning>
  Never commit your API key to version control. Use environment variables or the config file at `~/.naive/config.json`.
</Warning>

## How Responses Work

Every tool response includes **contextual hints** suggesting what to do next. For example, after a successful `send_email`:

```json
{
  "sent": true,
  "emailId": "em_abc123"
}

// ---
// Available actions:
// - list_inbox: Check for replies to this email
// - create_social_post: Cross-post this content to social media
// - generate_images: Create visual content for campaigns
```

These hints help your agent chain actions together without needing explicit instructions.

## Authentication

All primitives authenticate via the API key bound to a single **employee**. The key resolves to that employee and its parent **company** in Naive Core — you never need to pass `agent_id` or company identifiers. Email addresses, phone numbers, social accounts, payment cards, credentials, and credit balances are all scoped automatically.

## Orchestration Mode

The MCP server also supports **orchestration mode**, which expands the available tool surface with **44 additional platform orchestration tools** for managing companies, employees, tasks, deliverables, loops, and more. This mode uses **user-level keys** (`nv_pk_...`) instead of employee keys.

When using orchestration mode, `naive init` asks which mode you want:

- **Primitives mode** (default): Tools for doing work as an employee
- **Orchestration mode**: All primitives plus platform management tools

See the [Tool Catalog](/mcp/tool-catalog) for the full list of platform orchestration tools and the [Platform Orchestration Guide](/guides/platform-orchestration) for details on the two-tier architecture.
